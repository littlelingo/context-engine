# Skills and Agents: Full Lifecycle

## Skills

### Creation
Skills are NOT created dynamically at runtime. All 20 skill directories exist statically in `skills/`. New skills cannot be created on-the-fly — the skill set is fixed in the codebase.

### Population (Initial)
`/init` (`commands/init.md` lines 63-66) populates exactly two skills based on what the researcher agent discovers:
- `skills/testing-conventions/SKILL.md` — if test files detected
- `skills/api-conventions/SKILL.md` — if API/route files detected
All other skills remain as skeletons on first init.

### Skill Skeletons
Skills like `testing-conventions/SKILL.md` ship as comment-annotated templates:
```
<!-- Populated by /init. Grows via /learn and phase reflection. -->
## Test File Naming
<!-- e.g., [module].test.ts alongside source -->
```
They are filled in by /init or grow organically via /learn.

### Population (Ongoing)
Skills grow via two paths:
1. **Manual**: `/learn` routes knowledge to `.context/` files (not to `skills/` — see distinction below)
2. **Auto-capture**: `implementer` and `debugger` agents write to `.context/knowledge/` automatically when they discover library quirks, version pins, or integration insights

### The skills/ vs .context/ distinction
- `skills/` — framework-level domain expertise (universal patterns, language conventions, OWASP rules). Loaded on-demand by description matching.
- `.context/` — project-specific knowledge (this project's test patterns, this project's error history). Grown by agents and /learn.
`/learn` writes to `.context/`, not to `skills/`. Skills are NOT updated by /learn.

### Update
Skills are updated only by:
- Manual edits (developer adds project-specific rules to a skill)
- `/adapt` audit — but `/adapt` reads skills as standards to check against, does NOT write back to them
- `/update-arch` (`commands/update-arch.md`) updates `.context/architecture/` docs, not `skills/`

### Retirement
No automated retirement mechanism. Skills marked irrelevant in `.context/architecture/.skill-profile.json` (generated by `/init`) are deprioritized at runtime. Deprecated stacks require manual skill removal.

## Agents

### Definition
All 5 agents are static files in `agents/`:
- `researcher.md` — read-only, distilled summaries, memory: project
- `planner.md` — creates PRPs, memory: project
- `implementer.md` — executes PRP steps, writes to .context/knowledge/ automatically
- `reviewer.md` — code review only, no writes to .context/
- `debugger.md` — diagnoses bugs, writes to .context/errors/ automatically

All use YAML frontmatter with: name, description, tools (scoped), model (sonnet), memory (project).

### Static, Not Dynamic
Agents are NOT generated at runtime. `/init` does not create new agent files. `/adapt` does not create new agents. There is no mechanism to create a new agent role on-the-fly.

### Agent Memory
All agents use `memory: project` — they share project-scope persistent memory at `.claude/agent-memory/` (enforced by `guard-protected-files.sh`). Memory accumulates across sessions per role.

## Knowledge Capture Flow (the living layer)

The actual "living" part of the system is `.context/`, not `skills/` or `agents/`:

```
/implement (implementer agent) --> .context/knowledge/libraries/
                                   .context/knowledge/dependencies/PINS.md
                                   .context/patterns/CODE_PATTERNS.md
                                   .context/knowledge/LEARNINGS.md

/debug (debugger agent)        --> .context/errors/INDEX.md
                                   .context/errors/detail/ERR-NNN.md
                                   .context/knowledge/ (quirks discovered)

/learn (manual)                --> routes to appropriate .context/ subdirectory
                                   based on prefix or content inference

/update-arch                   --> .context/architecture/ (OVERVIEW, TECH_STACK, DIRECTORY_MAP)

/validate (post-phase)         --> triggers /update-arch if structural changes detected
```

## Key Insight: What Grows vs What Is Fixed

| Layer | Fixed/Dynamic | Changed By |
|-------|--------------|------------|
| `agents/` | Fixed (5 static files) | Manual dev edits only |
| `skills/` | Fixed (20 static dirs) | /init populates 2; manual edits otherwise |
| `.context/architecture/` | Dynamic | /init, /update-arch, researcher |
| `.context/patterns/` | Dynamic | implement/debug agents, /learn |
| `.context/errors/` | Dynamic | debugger agent, /learn |
| `.context/knowledge/` | Dynamic | implementer/debugger agents, /learn |
| `.context/features/` | Dynamic | /plan creates PRPs, /implement updates them |
| `.skill-profile.json` | Set once | /init; refreshable manually |
