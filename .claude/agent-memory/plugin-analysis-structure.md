# Context Engine Plugin Structure Analysis

## Plugin Entry Point and Identity

There is no compiled code. The plugin is a **pure markdown-and-shell artifact** — no TypeScript, no build step beyond file copying.

- **Manifest**: `.claude-plugin/plugin.json` — name `context-engine`, version `0.1.1`, `strict: true` in `marketplace.json`
- **Build script**: `build-plugin.sh` (root) — the sole "compilation" step. It copies files into `dist/context-engine-plugin/` and synthetically generates one file (the `context-engine-rules` skill).
- **Installed path**: `dist/context-engine-plugin/` is the deployable artifact. It mirrors the source layout exactly.

**Build script version** (`build-plugin.sh` line 14): `VERSION="0.0.5"` (hardcoded in script; manifest says `0.1.1` — these are out of sync).

## How the Plugin Generates MD Files

### 1. Build-time generation (build-plugin.sh)

The build script performs 11 steps, all `cp -r` except for two synthetic files:

| Step | What happens |
|------|-------------|
| 1 | Copy `.claude-plugin/` manifest |
| 2 | Copy `commands/` (18 `.md` files) |
| 3 | Copy `agents/` (5 `.md` files) |
| 4 | Copy `skills/` (20 skill dirs) |
| 5 | Copy `hooks/` (hooks.json + 9 shell scripts) |
| 6 | Copy `.mcp.json` |
| 7 | Copy `.claude/instructions/*.md` → `instructions/` |
| 8 | Copy `.context/*` → `context-templates/` (scaffolding templates for `/init`) |
| 9 | **Synthesize** `skills/context-engine-rules/SKILL.md` — frontmatter injected inline via heredoc, then `CLAUDE.md` is appended (`cat CLAUDE.md >> SKILL.md`) |
| 10 | Copy `README.md` + `docs/` |
| 11 | Inline-write `LICENSE` via heredoc |

**Key insight**: Step 9 is the only non-trivial generation. The `context-engine-rules` skill is assembled at build time by prepending YAML frontmatter and appending the repo's `CLAUDE.md`. This makes CLAUDE.md always-active context in any project that installs the plugin.

### 2. Runtime generation (/init command)

The `/init` command (`commands/init.md`) is where most MD file generation happens at runtime, inside user projects. It:

1. Creates `.context/` directory tree (10+ subdirectories)
2. Copies templates from `context-templates/` (installed at `${CLAUDE_PLUGIN_ROOT}/context-templates/`) into the user project's `.context/`
3. Delegates to the `researcher` subagent to scan the project
4. Generates 8 documents via LLM writing (not templates):
   - `architecture/OVERVIEW.md` — free-form LLM output
   - `architecture/TECH_STACK.md` — LLM output with detected stack
   - `architecture/DIRECTORY_MAP.md` — annotated tree
   - `patterns/CODE_PATTERNS.md` — detected patterns
   - `patterns/ANTI_PATTERNS.md` — empty skeleton
   - `errors/INDEX.md` — empty skeleton
   - `knowledge/LEARNINGS.md` — empty skeleton
   - `features/FEATURES.md` — empty feature index
5. Generates `.context/architecture/.skill-profile.json` — maps detected stack → relevant/irrelevant skills (used for runtime filtering)
6. Generates/merges `CLAUDE.md` in the user project (using the plugin's CLAUDE.md as a base template)
7. Optionally populates `skills/testing-conventions/SKILL.md` and `skills/api-conventions/SKILL.md` if test/API patterns are detected

## Output Format and Structure

### Commands (`commands/*.md`)
Plain markdown files. Structure:
```
# /[command-name] - [Title]
[1-2 sentence description]
## Process
[Numbered steps - LLM instructions]
## Rules
[Bullet constraints]
## User Input
$ARGUMENTS
```
No frontmatter. The `$ARGUMENTS` token is a placeholder that Claude Code substitutes at invocation time.

### Agents (`agents/*.md`)
YAML frontmatter + markdown body:
```yaml
---
name: [role]
description: [one-liner]
tools: [comma-separated tool allowlist]
model: sonnet
memory: project
---
[Role instructions in markdown]
```
The `tools:` field is a restrictive allowlist (e.g., researcher only gets Read, Glob, Grep, limited Bash patterns). The `memory: project` field wires agent memory to the project-scoped memory dir at `.claude/agent-memory/[role]/`.

### Skills (`skills/[name]/SKILL.md`)
YAML frontmatter + markdown body:
```yaml
---
description: [when this skill auto-loads and why]
---
[Skill content]
```
Some skills also have a `REFERENCE.md` (deeper detail — only loaded on demand). Skills with both files follow a Tier 1 (SKILL.md) / Tier 2 (REFERENCE.md) pattern.

Skills that have a REFERENCE.md: `mcp-tools`, `postgres-mcp`, `puppeteer`, `google-workspace`.

### Context Templates (`.context/` scaffolding)
Located in `context-templates/` in the plugin distribution. These are static markdown stubs with HTML comment placeholders:
```markdown
<!-- Generated by /init. Update with /update-arch. -->
## System Description
<!-- What does this project do? 2-3 sentences. -->
```
They are copied verbatim by `/init` — the LLM fills in the placeholders afterward.

### PRP Template (`context-templates/templates/PRP-TEMPLATE.md`)
Structured implementation plan format with: Status/Created/Complexity/Testing Strategy header fields, Requirements (must/nice-to-have/out-of-scope), Technical Approach table (file | action | description), Implementation Steps with checkboxes, Validation Checklist, Risks table, and a Metrics section (auto-populated by `/validate`).

### Capture Formats (`.claude/instructions/CAPTURE-FORMAT.md`)
Defines 6 rigid append formats for the auto-learning system:
- Error signatures → `.context/errors/INDEX.md` (ERR-NNN block format)
- Learnings → `.context/knowledge/LEARNINGS.md` (date + topic + 2-3 sentences)
- Patterns → `.context/patterns/CODE_PATTERNS.md` (name/context/example/rationale)
- Anti-patterns → `.context/patterns/ANTI_PATTERNS.md` (name/don't/do/why)
- Library quirks → `.context/knowledge/libraries/[name].md` (from TEMPLATE.md)
- Version pins → `.context/knowledge/dependencies/PINS.md` (package/version/reason/blocker)

## Token Efficiency Strategies

### 1. Progressive Disclosure (Skills)
Skills only load when Claude touches relevant files. The `description:` frontmatter field specifies trigger conditions (e.g., `Auto-loaded when working with .context/, agents, commands, or hooks`). This avoids loading all 20 skills upfront.

### 2. Skill Profile Filtering
`/init` generates `.context/architecture/.skill-profile.json` with `relevant_skills` and `irrelevant_skills` arrays. The `prompt-efficiency` skill (always loaded) instructs Claude to deprioritize irrelevant skills even if they auto-load.

### 3. Two-Tier Skill Loading
Skills with `REFERENCE.md` follow a strict rule: load only `SKILL.md` by default; read `REFERENCE.md` only when editing MCP configs or explicitly asked. This splits dense reference material from the activation trigger.

### 4. Delegation Without Context Duplication
`DELEGATION.md` instruction (`.claude/instructions/DELEGATION.md`): pass only task + file paths to subagents, never full context summaries. Agents read `.context/` independently. Commands don't repeat agent instructions.

### 5. Context Budget via Tool Count Proxy
`hooks/scripts/context-budget.sh` (PostToolUse on Read|Bash): increments a counter in `/tmp/.context-engine-tool-count`. Warns at 50 calls (prepare to `/clear`) and 80 calls (strongly recommend `/clear`). Proxy for context window fill.

### 6. Request Classification at Prompt Submission
`hooks/scripts/classify-request.sh` (UserPromptSubmit): regex-classifies each prompt as MINIMAL / STANDARD / HEAVY. MINIMAL skips delegation entirely. HEAVY prompts check budget before spawning Agent Teams. This prevents context overhead from trivial tasks.

### 7. Selective `.context/` Reading
Prompt-efficiency skill rule: "Read only the section needed from `.context/` files, not the full file." This is a behavioral instruction, not enforced by tooling.

### 8. CLAUDE.md Stays Lean
The CLAUDE.md has a core design principle: "This file stays lean - details live in `skills/` and `commands/`." The project knowledge table uses pointers to `.context/` files rather than inlining content.

## Hooks Architecture

Hooks are wired in `hooks/hooks.json` using `${CLAUDE_PLUGIN_ROOT}` for portability. All hooks are shell scripts.

| Event | Matcher | Script | Purpose |
|-------|---------|--------|---------|
| PreToolUse | Write/Edit/MultiEdit | `guard-protected-files.sh` | Block writes to protected paths |
| PreToolUse | Bash | `block-destructive.sh` | Block dangerous commands |
| PostToolUse | Write/Edit/MultiEdit | `auto-format.sh` | Auto-format after edits |
| PostToolUse | Read/Bash | `context-budget.sh` | Track tool call count, warn at thresholds |
| PreCompact | (all) | `preserve-context.sh` | Preserve PRP/branch/progress before compaction |
| Stop | (all) | `capture-learnings.sh` | Remind to capture learnings at session end |
| SubagentStop | (all) | `verify-agent-output.sh` | Verify agent output format |
| UserPromptSubmit | (all) | `session-track.sh` | Session boundary tracking |
| UserPromptSubmit | (all) | `classify-request.sh` | Inject complexity classification |

## Potential Issues / Notes

- **Version mismatch**: `build-plugin.sh` hardcodes `VERSION="0.0.5"` (line 14) while `plugin.json` says `0.1.1`. The build script's version string is cosmetic (only used in echo output), but it's stale.
- **No TypeScript/compilation**: There is no `src/`, `package.json`, or build toolchain. Everything is markdown + bash. If someone expects a compiled plugin, there is nothing to compile.
- **context-engine-rules skill is built at build time**: It doesn't exist in the source tree at `skills/context-engine-rules/SKILL.md` — only in `dist/`. The source only has the build script step that synthesizes it. Any edits to the plugin's CLAUDE.md flow through to this skill automatically on next build.
- **Skill profile filtering is behavioral, not enforced**: The `.skill-profile.json` mechanism relies on Claude following `prompt-efficiency` skill instructions. There is no hook that blocks irrelevant skill loading.
- **Context budget proxy is coarse**: The tool-count approach in `context-budget.sh` is a rough proxy. It counts Read+Bash calls, not actual token consumption.
